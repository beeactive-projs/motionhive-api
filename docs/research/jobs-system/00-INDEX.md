This is the single biggest latency win available: **moving these to a queue removes 200–800ms from p95 on auth/payment/session endpoints.**

### Stripe webhook handlers
- Entry point [webhook-handler.service.ts:120-241](../../../src/modules/payment/services/webhook-handler.service.ts#L120-L241).
- Idempotency via `webhook_event` table with UNIQUE on `stripe_event_id`.
- **Phase 2 (Connect onboarding) is fully wired**; Phase 3-5 (invoices, subscriptions, refunds/disputes/payouts) are stubs that delegate to service `syncFromWebhook` methods.
- Hard rule from CLAUDE.md (load-bearing): **every ORM call inside the handler must pass `{ transaction: tx }`**. Side effects (notifications, emails) happen **after commit**.
- Race condition exists: webhooks can arrive before the originating API call commits → status `ORPHANED`. Today there's no sweeper to retry these. (This is item #2 in the jobs-pending memory.)

## C. Reference architecture — `clym/events-api`

**Where**: `/Users/ionutbutnaru/Documents/clym/events-api` (ThorinJS, not NestJS — we extract patterns, not code).

### The pattern worth copying: hybrid persistence
Clym uses **two** queue mechanisms side-by-side and the split is principled:

- **BullMQ** for *event streams* — high-volume, real-time, stateless work. Two queues only (`ingest`, `publish_stream`) with 10 concurrency, exponential backoff, age-based cleanup.
- **Postgres `job` table** for *durable business workflows* — notifications, exports, scans. Statuses (NEW/PROCESSING/DONE/ERROR/RETRY), UID-based dedup with TTL, atomic `FOR UPDATE SKIP LOCKED` assignment, polled every 1.5s by support workers.
- **DB-driven cron** — `jobScheduler` table with `next_scheduled_at` precomputed, polled every 20s. *Not* BullMQ Repeat (so cron survives Redis restarts and is auditable in SQL).

The big structural lesson: **BullMQ is not the right tool for everything**. For business jobs where "what happened yesterday" matters, the DB job-table pattern is significantly better — and it composes with BullMQ rather than replacing it.

### The notification fan-out abstraction (the jewel)
Two-phase model worth copying near-verbatim:

1. **EventKind → Notification jobs** — when an event fires, look up `EventKind.notifications[]`. Each entry is `{ template_id, to_param: 'user.email' (JSONPath), base_template_id, segment_id }`. For each, create one job with structured `{ notify: [{to, template_id, blocks}] }` payload and a UID `${eventId}.${templateId}.${i}` for dedup.
2. **Job → channel adapter** — a separate "Partner Platform" service consumes the job, looks up the template, renders, and sends. **The producer never knows about SMTP / Twilio / FCM** — it only references template IDs and recipient parameters.

Templates have a hierarchy: `NotificationBaseTemplate` (shared header/footer) + `NotificationTemplate` (per-event-type, with `type: EMAIL|SMS|PUSH`). Channel preferences are *implied* in template type today; explicit per-user `channel_preferences` is the obvious extension.

This maps cleanly to NestJS:
```
EventKind.notifications → [NotificationJob] → ChannelAdapter
                                              ├─ EmailAdapter (Resend)
                                              ├─ PushAdapter  (web-push)
                                              ├─ SmsAdapter   (Twilio later)
                                              └─ InAppAdapter (Postgres + SSE)
```

### Idempotency — multi-layer
1. **Job UID** with `uid_expire` TTL (sha1 → unique_id, expires after window).
2. **`is_unique` flag** — broader: skip if any job with same group/name/run_type is in NEW/PROCESSING/RETRY.
3. **Source filtering** for multi-tenant worker affinity.
4. **In-batch dedup map** for broadcast fan-out (avoid sending twice in same batch).

### Patterns NOT to copy
- ThorinJS global singletons (`thorin.lib('queue')`) — use Nest DI.
- Implicit filesystem worker discovery (`thorin.loadPath(...)`) — explicit registration in modules.
- No metrics export — we should ship Prometheus from day 1 (file 07).
- Single cron poller without leader election — use Nest's distributed lock or stick with `@nestjs/schedule` and run the scheduler on one dyno.
- All-or-nothing retries — use typed errors (TemporaryError vs PermanentError) for retry decisions.

## E. The shape of the recommended architecture

Pulled from file 10 for quick reference (read file 10 for the reasoning):

```
┌────────────────────────────────────────────────┐
│ NestJS API (Railway)                           │
│  ├─ business code calls notify()               │
│  └─ webhook handlers write outbox + commit     │
└────────────────────────────────────────────────┘
                       ↓
┌────────────────────────────────────────────────┐
│ BullMQ on Redis (Railway addon ~$5/mo)         │
│  ├─ notifications queue                        │
│  ├─ payments queue                             │
│  ├─ sessions queue                             │
│  ├─ media queue                                │
│  ├─ analytics queue                            │
│  ├─ auth queue                                 │
│  └─ maintenance queue                          │
└────────────────────────────────────────────────┘
                       ↓
┌────────────────────────────────────────────────┐
│ Workers (same dyno → eventually separate)      │
│  ├─ email_send → Resend                        │
│  ├─ push_send  → web-push lib (VAPID)          │
│  ├─ sms_send   → Twilio (when needed)          │
│  └─ in_app_create → Postgres + Redis pub/sub   │
│                            ↓                    │
│                       SSE stream → frontend     │
└────────────────────────────────────────────────┘
```

**Three decisions that shape everything** (file 10 §"three decisions"):
1. **Queue runtime**: BullMQ (vs pg-boss) — pick BullMQ, migrate `@nestjs/bull` → `@nestjs/bullmq` in the same PR.
2. **Notification system**: DIY (vs Novu/Knock SaaS) — DIY is ~3.5 weeks one-time vs $250–2000/mo forever.
3. **Worker process model**: same-process day 1, separate process when growth pressure shows up — don't over-engineer the bootstrap.
